1. File System Management

1.1 Basic File Operations

Problem 1:

Find all files in '/var/log' directory which are older than 30 days and move to 'tmp' directory.

Solution:

find /var/log -type f -mtime +30 -exec mv {} /tmp/ \;

Use Case:

Automated log rotation and cleanup to free up disk space. Useful for maintaining log directories.

Problem 2:

Use less and grep command to print only August 25 data from the log file /var/log/messages.

Solution:

grep "^Aug 25" /var/log/messages | less

OR

less /var/log/messages | grep "Aug 25"

Problem 3:

Using grep command only and print 10 lines from a log file such as /var/log/messages.

Solution:

head -10 /var/log/messages

OR using grep:

grep -m 10 "." /var/log/messages

Problem 4:

Print log lines which have the word 'product' from the file /var/log/messages.

Solution:

grep -i 'product' /var/log/messages

-i flag makes the search case-insensitive.

Problem 5:

Search 'Product' word in the file and wherever it is found, print the below 2 lines from the line the word is found.

Solution:

grep -A 2 -i 'Product' /var/log/messages

-A 2 shows 2 lines After each match.

Problem 6:

What is the purpose of 'execute' permissions to a folder?

Solution:

Execute (x) permission on a directory allows:

  • Changing into the directory using cd
  • Accessing files and subdirectories within (if you know their names)
  • Viewing directory metadata (combined with read permission)

Without execute permission: Users cannot access directory contents even if they have read permission.

Problem 7:

I want to run one utility with root privileges?

Solution:

sudo /path/to/utility

OR for specific user context:

sudo -u root /path/to/utility

OR to run as another user:

sudo -u himanshu /path/to/somefile

1.2 Special File Permissions

SUID (Set User ID):

chmod u+s /path/to/file

File executes with owner's privileges regardless of who runs it.

Example: /usr/bin/passwd

SGID (Set Group ID):

chmod g+s /path/to/directory

New files created in the directory inherit the directory's group ownership.

Sticky Bit:

chmod +t /path/to/directory

Only file owner (and root) can delete files in the directory.

Example: /tmp directory

1.3 ACL (Access Control Lists) Scenario

Scenario Requirements:

  1. Project Managers (group proj_managers) - read, write, and execute on all project directories
  2. Developers (group developers) - read and execute only (no delete/modify)
  3. QA Engineers (group qa_engineers) - read-only access to project_alpha only
  4. User alice - read, write, execute on project_beta only
  5. Default ACLs for inheritance on new files/directories

Directories: /projects/project_alpha, /projects/project_beta, /projects/project_gamma

Complete ACL Solution:

# 1. Project Managers - full access setfacl -m g:proj_managers:rwx /projects setfacl -R -m g:proj_managers:rwx /projects/* # 2. Developers - read/execute only setfacl -m g:developers:r-x /projects setfacl -R -m g:developers:r-x /projects/* # 3. QA Engineers - specific directory only setfacl -m g:qa_engineers:r-x /projects/project_alpha # 4. Alice - specific access setfacl -m u:alice:rwx /projects/project_beta # 5. Default ACLs for inheritance setfacl -d -m g:proj_managers:rwx /projects setfacl -d -m g:developers:r-x /projects

Verify ACLs:

getfacl /projects/project_alpha

Remove ACL entry:

setfacl -x u:alice /projects/project_beta

1.4 File System Attributes

Scenario:

Legal department's documents need protection from accidental deletion. Apply append-only attribute.

Solution:

# Apply append-only attribute chattr +a /company_data/legal/* # Verify attributes lsattr /company_data/legal/ # Remove attribute chattr -a filename

Common attributes:

  • +a - Append only (can't delete/modify, only append)
  • +i - Immutable (cannot be changed at all)
  • +A - No atime updates (access time not updated)

2.4 Specific User Management Tasks

Password Policy:

Set password change every 90 days for user mike_b.

Solution:

sudo chage -M 90 mike_b

Directory Restriction:

Set permissions on /projects/alpha/ so only Mike can read, write, and execute.

Solution:

sudo chmod 700 /projects/alpha/

OR set owner and then permissions:

sudo chown mike_b /projects/alpha/ sudo chmod 700 /projects/alpha/

User Switching:

Mike needs to temporarily assume sara_c's identity.

Solution:

# Switch to sara_c user (requires sara_c's password) su - sara_c # OR run single command as sara_c sudo -u sara_c /usr/local/bin/myscript.sh

Sudo Privileges:

Add Mike to sudo group with limited package installation privileges.

Solution:

# Add to sudo group sudo usermod -aG sudo mike_b # OR specific sudoers entry echo "mike_b ALL=(ALL) NOPASSWD: /usr/bin/apt install *" | sudo tee -a /etc/sudoers

Package installation example:

sudo apt install nginx

Service Control Privileges:

Add user with privileges to start/stop services.

Solution:

echo "username ALL=(ALL) NOPASSWD: /bin/systemctl start *, /bin/systemctl stop *, /bin/systemctl restart *" | sudo tee -a /etc/sudoers

3. Process Management

Question 1:

How many signals are there in the kill command?

Solution:

64 signals (1-64)

kill -l # List all signals

Common Signals:

  • SIGTERM (15): Graceful termination
  • SIGKILL (9): Force kill (cannot be caught/ignored)
  • SIGSTOP (19): Pause process
  • SIGCONT (18): Continue paused process
  • SIGHUP (1): Hangup (reload configuration)

Question 2:

Zombie process identification and handling.

Solution:

# Identify zombie processes ps aux | grep 'Z' ps -eo pid,ppid,state,cmd | grep -E 'Z|DEFUNCT' # Find parent of zombie ps -eo pid,ppid,cmd | grep [parent_pid]

Solution: Kill the parent process or reboot system.

Question 3:

List multi-threaded processes.

Solution:

# Show threads (LWP - Light Weight Processes) ps -eLf # Show threads in top top -H # Count threads per process ps -eo nlwp,pid,cmd | sort -rn | head

Question 4:

Process hierarchy with PIDs using pstree.

Solution:

pstree -p # Show PIDs pstree -p -u # Show PIDs and users pstree -p -a # Show PIDs and arguments pstree -p [pid] # Show subtree for specific process

Question 5:

Check memory utilization and explain vmstat output.

Solution:

vmstat 2 5 # Every 2 seconds, 5 times

Key vmstat output fields:

  • r: Run queue (processes waiting for CPU)
  • b: Blocked processes
  • swpd: Swap used (KB)
  • free: Free memory (KB)
  • buff: Buffer cache memory (KB)
  • cache: Page cache memory (KB)
  • si: Swap in from disk (KB/s)
  • so: Swap out to disk (KB/s)
  • us: User CPU time (%)
  • sy: System CPU time (%)
  • id: Idle CPU time (%)

Question 6:

Explain output of top and htop commands.

Solution:

Top command sections:

  1. Uptime & Load Average: System uptime and 1/5/15 minute load averages
  2. Tasks: Total, running, sleeping, stopped, zombie processes
  3. CPU Usage: us(user), sy(system), ni(nice), id(idle), wa(wait), hi(hardware interrupt), si(software interrupt), st(steal)
  4. Memory: Total, free, used, buff/cache
  5. Swap: Total, free, used, available
  6. Process List: PID, USER, PR, NI, VIRT, RES, SHR, S, %CPU, %MEM, TIME+, COMMAND

Load Average Interpretation:

  • < 1.0 per CPU core: System is idle
  • = 1.0 per CPU core: System fully utilized
  • > 1.0 per CPU core: Processes waiting for CPU

htop advantages: Color-coded, vertical/horizontal scrolling, tree view, mouse support.

Question 11:

How to kill a particular process?

Solution:

Question 12:

Explain load average parameter in top command.

Solution:

Load Average: Shows system load over 1, 5, and 15 minutes.

Interpretation: Value represents average number of processes in run queue.

  • On single-core CPU: 1.00 = 100% utilization
  • On quad-core CPU: 4.00 = 100% utilization
  • < CPU cores: System underutilized
  • = CPU cores: Fully utilized
  • > CPU cores: Processes waiting
# Check number of CPU cores nproc grep -c ^processor /proc/cpuinfo

Question 13:

Types of processes.

Solution:

  1. Foreground Processes: Interactive, controlled by terminal
  2. Background Processes: Run with &, not tied to terminal
  3. Daemon Processes: System/services, run in background
  4. Zombie Processes: Completed but not reaped by parent
  5. Orphan Processes: Parent died, adopted by init (PID 1)
  6. User Processes: Started by users
  7. System Processes: Started by kernel

Using ps command, display pid, user, cpu, cmd.

Question 8:

Buffer vs Cache explanation and buffer overflow attack.

Solution:

Buffer: Temporary storage area for data in transit between processes/devices.

Cache: Stores frequently accessed data for faster retrieval.

Aspect Buffer Cache
Purpose Hold data during transfer Speed up data access
Lifespan Short-term Long-term
Location RAM RAM, CPU (L1/L2/L3)
Example Keyboard buffer, network buffer CPU cache, DNS cache, web cache

Buffer Overflow Attack: When more data is written to a buffer than it can hold, overwriting adjacent memory.

Prevention: Input validation, bounds checking, address space layout randomization (ASLR).

Additional Question:

Types of partitions - need for logical partitions even with primary partitions.

Solution:

Partition Types:

  1. Primary Partitions: Max 4 per disk, bootable, directly accessible
  2. Extended Partition: Special type that acts as container for logical partitions
  3. Logical Partitions: Created inside extended partition, unlimited number

Why logical partitions needed:

  • MBR limitation: Only 4 primary partitions allowed
  • Extended partition + logical partitions bypass this limit
  • Allows more than 4 partitions on same disk
  • Logical partitions numbered from 5 upwards
# View partition table sudo fdisk -l lsblk parted -l

5. Disk & Memory Management

Question 1:

Swap memory - How to make swap memory permanent? Update /etc/fstab file with explanation of each column.

Solution:

# Create swap file sudo fallocate -l 8G /swapfile sudo chmod 600 /swapfile sudo mkswap /swapfile sudo swapon /swapfile # Make permanent in /etc/fstab echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

/etc/fstab columns explained:

  1. Device/File: /swapfile (or /dev/sda2)
  2. Mount point: none (swap doesn't have mount point)
  3. Filesystem type: swap
  4. Options: sw (swap specific options)
  5. Dump: 0 (not backed up by dump utility)
  6. Pass: 0 (not checked by fsck at boot)

Question 2:

Add 8 GB of swap in your system.

Solution:

# Step-by-step 8GB swap creation sudo fallocate -l 8G /swapfile # OR if fallocate fails: sudo dd if=/dev/zero of=/swapfile bs=1G count=8 sudo chmod 600 /swapfile sudo mkswap /swapfile sudo swapon /swapfile # Verify swapon --show free -h cat /proc/swaps

Question 3:

Check your disk usage.

Solution:

# Human readable df -h # Inode usage df -i # Filesystem type included df -hT # Specific directory df -h /home # All mounted filesystems mount | column -t

Question 4:

Partition is 100% utilized. Clean up and find largest files.

Solution:

# 1. Identify large files (>100MB) find / -type f -size +100M 2>/dev/null | head -20 # 2. Find top 10 largest directories du -ah / 2>/dev/null | sort -rh | head -10 # 3. Check specific directory du -sh /home/* # 4. Clean package cache sudo apt clean # Debian/Ubuntu sudo yum clean all # RHEL/CentOS sudo dnf clean all # Fedora # 5. Clear old logs sudo journalctl --vacuum-time=30d # 6. Remove old kernels (Ubuntu) sudo apt autoremove --purge # 7. Find and delete core dumps find / -name "core" -type f -delete 2>/dev/null find / -name "*.core" -type f -delete 2>/dev/null

Question 5:

How to check performance of your virtual memory.

Solution:

# 1. vmstat for virtual memory stats vmstat 2 10 # 2. sar for historical data sar -r 2 10 # 3. Check swap usage swapon --show free -h # 4. Check page faults sar -B 2 10 # 5. Check memory pressure cat /proc/pressure/memory # 6. Check swap in/out rates vmstat -s | grep -E "swap|page" # 7. Monitor with top/htop htop # Check swap bar and memory usage

Key metrics to monitor:

  • Swap usage (should be minimal)
  • Page in/out rates (high rates indicate thrashing)
  • Free memory vs cache/buffer memory
  • Swap I/O (si/so in vmstat)

Additional Question:

What is cylinder in disk drive?

Solution:

Cylinder: Set of tracks at same position on all platters of a hard disk.

CHS Addressing (Legacy):

  • Cylinder: Vertical position of read/write heads
  • Head: Which platter surface
  • Sector: Specific section on track

Modern systems: Use LBA (Logical Block Addressing) instead of CHS.

7.3 System Health Monitoring Script

Complete Monitoring Script:

#!/bin/bash
# system_health_check.sh

echo "========================================="
echo "    SYSTEM HEALTH CHECK - $(date)"
echo "========================================="
echo ""

echo "1. SYSTEM UPTIME & LOAD:"
uptime
echo ""

echo "2. MEMORY USAGE:"
free -h
echo ""

echo "3. DISK USAGE:"
df -h | grep -v tmpfs
echo ""

echo "4. TOP 5 CPU PROCESSES:"
ps aux --sort=-%cpu | head -6
echo ""

echo "5. TOP 5 MEMORY PROCESSES:"
ps aux --sort=-%mem | head -6
echo ""

echo "6. NETWORK CONNECTIONS:"
ss -t -a | head -10
echo ""

echo "7. LOGGED IN USERS:"
who
echo ""

echo "8. RECENT AUTH FAILURES:"
tail -20 /var/log/auth.log | grep -i "fail" | tail -5
echo ""

echo "========================================="
echo "Check completed at: $(date)"
echo "========================================="

Make executable and run:

chmod +x system_health_check.sh ./system_health_check.sh

Important Notes:

  • Always test commands in non-production environment first
  • Use sudo only when necessary
  • Backup configuration files before making changes
  • Monitor logs regularly: /var/log/syslog, /var/log/auth.log, application logs
  • Implement proper backup strategies for critical data
